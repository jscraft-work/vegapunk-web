# vegapunk 기획서

> 개인 지식베이스 기반 AI 대화 비서 — "제2의 뇌(second brain)"

## 1. 개요

vegapunk는 **내가 쌓아둔 지식 노트를 근거로 대답해주는 개인용 AI 채팅 서비스**다. 사용자는 평소처럼 비서와 대화하고, 비서는 답할 때 **사용자가 직접 저장해둔 노트**를 찾아 근거로 활용한다(RAG). 또한 나눈 대화를 **주제별 노트로 자동 정리(distill)**해 지식을 계속 불려나간다. 노트끼리는 위키처럼 `[[링크]]`로 엮이고 태그로 분류된다.

목표는 **대화 품질** — 흩어진 내 지식을 정확히 찾아, 맥락을 잃지 않고, 군더더기 없이 답하는 것. (다수 사용자 확장은 이번 범위 밖, 향후 과제.)

**비용 제약**: LLM은 호스트의 기존 래퍼(openclaw)를 재사용한다(추가 비용 0). 임베딩은 로컬 모델(무비용), 저장소는 **기존에 운영 중인 Postgres를 재사용**(추가 비용 0).

## 2. 용어 정의 (이 문서를 처음 보는 사람 기준)

| 용어 | 뜻 |
|------|----|
| **노트(Note)** | 지식 한 토픽을 담은 위키 페이지. 제목 + 마크다운 본문. |
| **위키링크** | 본문 속 `[[다른 노트 제목]]`. 노트끼리의 연결. |
| **태그** | 노트 분류 라벨. 한 노트에 여러 개. |
| **대화(Conversation)** | 사용자와 비서가 주고받은 한 세션. |
| **distill(증류)** | 대화에서 보관 가치 있는 내용을 비서가 노트로 정리하는 기능. |
| **RAG** | 답하기 **전에** 노트를 검색해, 관련 내용을 비서에게 자료로 같이 건네주는 방식. |
| **청크(Chunk)** | 노트 본문을 검색용으로 잘게 나눈 문단 단위. |
| **임베딩(Embedding)** | 글의 "의미"를 숫자 벡터로 바꾼 것. 뜻이 비슷하면 벡터도 가까움. (로컬 계산 → 무비용) |
| **하이브리드 검색** | "글자 일치 검색"과 "의미 검색"을 둘 다 돌려 합치는 방식. |
| **RRF** | 두 검색의 순위표를 합산해 최종 순위를 내는 방법(Reciprocal Rank Fusion). 양쪽에서 다 상위인 게 좋은 결과. |
| **다시쓰기(query rewrite)** | "그거 왜?" 같은 맥락 의존 질문을, 검색 가능한 완결 문장으로 바꾸는 단계. |
| **openclaw** | 호스트의 LLM 호출 래퍼. **메시지 하나를 받아 답 하나를 주는** 단순 입구. 추가 비용 없음. |

## 3. 핵심 기능

1. **AI 채팅** — 질문하면 내 노트를 근거로 답한다. 후속 질문("그거 왜?")도 맥락을 이어 이해한다.
2. **지식 검색** — 노트를 글자/의미로 하이브리드 검색.
3. **지식 저장(distill)** — 대화를 주제별 노트로 자동 정리 → 검토 후 저장.
4. **노트 관리** — 위키링크/태그로 연결, 백링크·고아노트 확인, 수정 이력/되돌리기.
5. **로그인** — OAuth로 접근 보호.

## 4. 사용자 시나리오

**A. 대화하며 답 얻기**
"연봉협상 어떻게 했었지?" → 노트 검색 → "연봉협상" 노트 근거로 답변 → 출처 표시.
이어서 "그거 왜 그렇게 정했어?" → '그거=연봉협상'임을 맥락으로 파악(다시쓰기) → 다시 검색해 답변.

**B. 대화를 지식으로 저장**
대화 후 "지식으로 저장" → 비서가 주제별 노트 초안 생성 → 검토/수정 → 저장(기존 노트면 병합).

**C. 지식 탐색**
태그 필터, 노트 열람, `[[링크]]` 이동, 백링크 확인.

## 5. 시스템 구성

```
[브라우저 SPA]  ──HTTP/SSE──>  [웹 서버 (FastAPI)]
                                   │
        ┌──────────────┬──────────┴───────────┬─────────────┐
   [검색엔진]      [임베딩(로컬)]          [LLM = openclaw]   [저장소(Postgres)]
  글자+의미 융합   노트→벡터 변환        한 메시지 in→답 out   노트/대화/그래프
```

- **임베딩**: 로컬 fastembed(multilingual-e5-large, 1024차원). **무비용.**
- **LLM**: openclaw 재사용. **session_id 안 씀(무상태)**, 메시지를 앱이 직접 조립.
- **저장소**: Postgres + pgvector(의미검색) + pg_bigm(글자검색, 2-gram CJK). 기존 운영 DB 재사용, 백업·운영 도구 그대로.

## 6. LLM 연동 (핵심 설계)

### 원칙
openclaw는 **메시지 하나 → 답 하나**짜리 무상태 입구로만 쓴다. **대화 기억은 앱이 소유한다.** session_id는 사용하지 않는다.
→ 이렇게 해야 매 턴 RAG를 자유롭게 갈아끼우고(옛 RAG 누적 방지), 대화 기억을 직접 통제할 수 있다.

### openclaw의 제약
openclaw는 `messages` 배열이 아니라 **`prompt` 문자열 하나**만 받는다. 따라서 우리가 조립한 대화(지시 + 기억 + RAG + 질문)를 **한 덩이 문자열로 합쳐서** 보낸다. 역할 구분은 텍스트 안에 `사용자:`/`비서:` 표시로 대신한다.

### 매 턴 파이프라인
```
1. 질문 받기
2. [이전 대화 있으면] 다시쓰기 (빠른 등급 low)
      입력: 요약 + 직전 1~2턴 + 질문   → 검색쿼리 한 줄
      [첫 질문이면] 생략, 질문 그대로 사용
3. 검색: 검색쿼리 임베딩 → FTS+벡터 → RRF → 그래프확장 → 상위 ~5청크
4. 답변 조립: 지시 + 요약 + 최근 N턴(원문) + RAG + 원문질문
5. 답변 호출 (session_id 없음) → 스트리밍
6. 저장: user/assistant 메시지 + citations
7. (백그라운드) 창 넘쳤으면 요약 증분 갱신   ← 답변을 막지 않음
```

**LLM 호출**: 다시쓰기 1회(`low`, 가벼움) + 답변 1회. 요약은 가끔, 백그라운드.

### 다시쓰기를 채택한 이유
- "그거 왜?"를 그대로 검색하면 빈손으로 돌아옴 → 후속 질문 품질의 핵심 레버.
- 비용 0(openclaw), 빠른 등급이면 지연 ~0.3~0.5초로 작음.
- production RAG 챗봇의 표준 단계(검색어 품질이 검색 결과를 좌우).
- 대안(이어붙이기 꼼수)은 빠르지만 검색이 거칠어 부정확 → 정확도 우선이라 다시쓰기 채택.

### 검색용 vs 답변용 — 넣는 양이 다름 (혼동 방지)
| | 검색용 | 답변용 |
|--|--|--|
| 크기 | **작게** | 크게 |
| 내용 | 다시쓴 한 줄 쿼리 | 요약 + 최근 N턴 + RAG + 질문 |
| RAG | 없음(검색 전) | 있음 |

검색은 초점이 좁아야 잘 된다(긴 덩어리를 임베딩하면 벡터가 뿌예져 정확도↓). LLM 답변은 긴 맥락을 잘 다루므로 풍부하게.

### 조립 형식 (답변용, 한 덩이 문자열)
```
[지시]   너는 배종수의 비서다... (출처표기·헤지 금지 등)
[지금까지 요지]   ← 대화가 길 때만. 오래된 부분 요약
[최근 대화]       ← 최근 N턴 원문 (사용자:/비서:)
[참고 자료]       ← 이번 질문용 RAG만. 옛 RAG는 안 넣음
[질문]            ← 원문 질문 (다시쓴 건 검색용일 뿐, 여기엔 원문)
```

## 7. 대화 기억 / 압축 (효율화 확정)

### 핵심 3가지

**① 증분 (전체 재요약 절대 안 함)**
```
요약 입력 = 기존 요약  +  이번에 새로 밀려난 턴들만
요약 출력 = 갱신된 요약
```
이미 요약된 옛것은 다시 안 봄 → 입력 크기 항상 작고 고정. **전체 대화를 통째로 보내지 않는다**(요약의 존재 이유 자체가 그것).

**② 배칭 (매 턴 안 함)**
```
목표 창: 최근 6턴 원문 유지
12턴까지는 그냥 둠 → 12턴 넘으면 오래된 6턴을 한 번에 접기 → 다시 6턴
```
→ 요약 호출이 ~6턴에 1번 꼴로 드뭄.

**③ 백그라운드 (응답 안 막음)**
요약은 *다음* 턴용이라 늦어도 됨 → **답을 먼저 보내고** 뒤에서 갱신. 체감 속도 영향 0.

### 불변식
> 모든 과거 턴은 `summary`(접힌 것) 또는 `최근 N턴`(원문) 중 **하나에 반드시** 존재.
> → 창에서 **버리기 전에 먼저 요약에 접는다.** (이 순서만 지키면 맥락 구멍 없음)

### 다시쓰기·요약의 맥락 커버
- 초반 "그거": 아직 밀려난 턴이 없으니 **원문 창**이 직접 커버(요약 불필요).
- 후반 "그거": 밀려난 것은 이미 **요약**에 들어가 있어 커버.
- 드문 깊은 참조(요약에서 손실): 다시쓰기든 이어붙이기든 못 풂. 단, 답변 호출엔 요약+최근턴이 들어가므로 **대화 이해 자체는 살아있음**(검색 근거만 빗나감).

## 8. RAG / 검색 설계

- **청크 단위 검색**: 노트를 문단으로 쪼개 임베딩. 긴 노트에서도 관련 문단만 정확히.
- **하이브리드 + RRF 융합**: 글자검색(FTS5 trigram, 한글 대응) + 의미검색(벡터) 각각 순위 → 합산.
- **그래프 확장**: 검색으로 찾은 노트의 `[[링크]]` 이웃을 1단계 끌어와 컨텍스트 보강.
- **추리기**: 후보를 넉넉히(~20) 모은 뒤 상위 ~5개만 비서에게 전달.

## 9. 데이터베이스 설계 (Postgres + pgvector + pg_bigm)

```sql
-- ── 확장 (1회) ─────────────────────────
CREATE EXTENSION IF NOT EXISTS vector;    -- pgvector: 의미검색(벡터)
CREATE EXTENSION IF NOT EXISTS pg_bigm;   -- 글자검색(2-gram, 한글 등 CJK에 강함 — 2글자 쿼리도 인덱스 사용. 트리거 불필요)

-- ── 노트 ───────────────────────────────
CREATE TABLE notes (
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title      TEXT UNIQUE NOT NULL,
  body       TEXT NOT NULL,                       -- 현재 본문(빠른 조회용)
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE note_versions (                       -- 수정 이력(되돌리기)
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  note_id    BIGINT NOT NULL REFERENCES notes ON DELETE CASCADE,
  body       TEXT NOT NULL,
  source     TEXT,                                 -- 'manual' | 'distill'
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- ── 검색 (청크 + 의미 + 글자) ───────────
CREATE TABLE chunks (                              -- 본문을 문단으로 쪼갠 검색 단위
  id        BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  note_id   BIGINT NOT NULL REFERENCES notes ON DELETE CASCADE,
  ord       INTEGER NOT NULL,                      -- 노트 내 순서
  text      TEXT NOT NULL,
  embedding vector(1024)                           -- 의미검색용. e5 "passage: " 임베딩
);
-- 의미검색: 코사인 HNSW (sqlite-vec의 vec0를 대체)
CREATE INDEX idx_chunks_embedding ON chunks USING hnsw (embedding vector_cosine_ops);
-- 글자검색: bigram GIN (FTS5 trigram을 대체. LIKE '%...%' 가속 + bigm_similarity 랭킹. INSERT/DELETE 시 자동 유지 → 트리거 불필요)
CREATE INDEX idx_chunks_bigm ON chunks USING gin (text gin_bigm_ops);
CREATE INDEX idx_chunks_note ON chunks(note_id);

-- ── 그래프 (연결) ──────────────────────
CREATE TABLE edges (
  src_note  BIGINT NOT NULL REFERENCES notes ON DELETE CASCADE,
  dst_title TEXT NOT NULL,            -- 링크 대상 제목([[제목]]의 텍스트). 항상 존재 → PK 일부
  dst_note  BIGINT REFERENCES notes ON DELETE CASCADE,  -- 해석 캐시: 그 제목의 노트 있으면 id, 없으면 NULL(미해결)
  kind      TEXT NOT NULL DEFAULT 'wikilink',
  PRIMARY KEY (src_note, dst_title, kind)
);
CREATE INDEX idx_edges_dst ON edges(dst_note);   -- 백링크 빠르게
-- 노트 생성/삭제 시 같은 제목의 dst_title을 가진 edges의 dst_note를 채우거나 비움(미해결 링크 자동 해결)

-- ── 태그 ───────────────────────────────
CREATE TABLE tags (
  id   BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  name TEXT UNIQUE NOT NULL
);
CREATE TABLE note_tags (
  note_id BIGINT NOT NULL REFERENCES notes ON DELETE CASCADE,
  tag_id  BIGINT NOT NULL REFERENCES tags  ON DELETE CASCADE,
  PRIMARY KEY (note_id, tag_id)
);

-- ── 대화 ───────────────────────────────
CREATE TABLE conversations (
  id                  BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  title               TEXT,
  summary             TEXT,           -- ★ 오래된 대화의 요지(증분 압축)
  summary_upto_msg_id BIGINT,         -- ★ 어디까지 요약에 접혔는지 경계
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- session_id 없음! (openclaw 무상태로 쓰고, 기억은 앱이 소유)

CREATE TABLE messages (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  conv_id     BIGINT NOT NULL REFERENCES conversations ON DELETE CASCADE,
  role        TEXT NOT NULL,          -- 'user' | 'assistant'
  content     TEXT NOT NULL,          -- 원문만 저장(RAG·요약 안 섞음)
  sent_prompt TEXT,                   -- (디버그) 실제 보낸 한 덩이 문자열
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_messages_conv ON messages(conv_id, id);

CREATE TABLE message_citations (      -- 이 답변이 근거로 쓴 노트 (note_id 기준 → 재인덱싱에도 안정)
  message_id BIGINT NOT NULL REFERENCES messages ON DELETE CASCADE,
  note_id    BIGINT REFERENCES notes ON DELETE SET NULL,
  score      REAL,
  PRIMARY KEY (message_id, note_id)
);

-- ── 사용자 (인증, 최소) ────────────────
CREATE TABLE users (
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  email      TEXT UNIQUE NOT NULL,
  name       TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

> **스키마 운영**: 마이그레이션은 `migrations/NNNN_*.sql` 순번 파일로 관리(앱 부팅 시 `schema_migrations` 테이블 기준으로 미적용분만 실행). pgvector·pg_bigm 확장 생성 권한이 필요하므로 최초 1회는 superuser 또는 확장 사전설치 환경에서 적용.

### 기존 구현 대비 변화
| | 기존 | 기획 |
|--|--|--|
| 검색 단위 | 노트 통째 | **청크(문단)** |
| 대화 기억 | openclaw session_id 위임 | **앱이 소유(summary + messages)** |
| RAG | 매 턴 누적 | **최신 턴만, 안 쌓음** |
| 후속질문 | 원문 그대로 검색 | **다시쓰기로 검색쿼리 생성** |
| 연결 | 본문 regex 재파싱 | **edges 테이블** |
| 태그 | 콤마 문자열 | **정규화 + 그래프화** |
| 이력 | 없음 | **note_versions** |
| 출처 | 메시지에 JSON | **message_citations 정규화** |

## 10. 대화 레이어 파라미터 (초기값, 돌려보며 조정)

| 항목 | 값 |
|--|--|
| 최근 원문 유지 | 6턴 (또는 ~3000토큰) |
| 요약 트리거 | 12턴 초과 |

## 11. 기술 스택

## 12. distill (지식 저장) 설계

### 후보(candidate)
distill은 대화를 읽고 **저장 제안 = 노트 후보**를 만든다. 후보 = 아직 DB에 없는 `{title, body(markdown), tags[]}` 초안. 주제가 여럿이면 여러 후보로 분할, 보관 가치 없으면 빈 배열.

**distill 프롬프트 골자**
- 입력: 대화 전체 + 기존 노트 제목 목록 + 기존 태그 목록.
- 주제별로 후보 분할. 인사·잡담은 제외(빈 배열).
- 각 후보 = `{title, body, tags[]}`. 관련 노트는 본문에 `[[제목]]` 링크.
- 태그는 기존 태그 우선 재사용(난립 방지).
- JSON 배열로만 응답. 앞뒤 잡소리 허용하고 `[...]`만 추출; body에 실제 줄바꿈이 들어오므로 **관대한 JSON 파싱**(엄격 모드 금지).

후보들은 검토 UI로 사용자에게 보여주고, 승인/수정 후 저장(저장 = 본문 확정 + 13장 인덱싱).

### 병합 대상 찾기 (각 후보별)
"제목 일치"만으론 부족(같은 주제, 다른 제목). 검색엔진을 재사용한다. **후보 본문을 통째로 임베딩하지 않는다** — 길면 벡터가 뿌예져 매칭 실패(검색 쿼리와 동일한 함정).

1. **제목 신호(싸고 강력)**: 후보 제목이 기존 제목과 정확/근접 일치 → 강한 병합 신호. 제목만 임베딩(짧아서 또렷)해 유사 제목도 탐색.
2. **청크 대 청크(정밀)**: 후보 본문을 **청크로 쪼개**(어차피 인덱싱 때 쪼갬), 각 청크로 기존 청크를 의미검색 → 매칭이 **어느 노트에 몰리는지** 집계.
3. **판정**: 매칭 집중도(+제목 가중치)가 임계 이상 → "기존 [X]에 병합" 제안. 임계 미만 → **새 노트 기본**(확신 없으면 안 합침 — 오병합이 중복보다 나쁨).

사용자가 검토에서 최종 결정: `[기존 X에 병합] [새 노트로] [다른 노트 선택] [버림]`.
**후보 1개 → 대상 1개 원칙** (두 노트에 걸치면 distill이 덜 쪼갠 것 → 후보 분할을 유도).

### 병합 방식
**이어붙이기(append) 안 함** — 중복·모순이 쌓여 노트가 썩고, 그 청크가 RAG에 들어가 답 품질을 떨어뜨린다.

**LLM 통합(재작성)이 기본**, 위험(손실·환각)은 3중 안전망으로 차단:
1. **보존 규칙 프롬프트로 통합본 생성**
   - 기존 정보는 모순되지 않는 한 **모두 보존**(임의 삭제 금지)
   - 추가 내용은 **관련 위치에 통합**(끝에 붙이기 금지)
   - 중복은 병합, 같은 말 반복 금지
   - 모순 시 **최신 채택 + 표시**: 예) `연봉 6천 (이전 5천에서 변경)`
   - 마크다운 구조·`[[링크]]`·표 보존
   - 통합본 본문만 출력
2. **diff 검토** — 옛 본문 vs 통합본을 사용자가 보고 승인/수정 후 저장.
3. **버전 백업** — 저장 시 `note_versions`에 이전 본문 적재 → 잘못되면 복구.

**긴 노트**: 통째 재작성은 토큰·위험↑ → 영향받는 **섹션(`##` 헤딩)만** 통합, 나머지 보존. 짧은 노트는 통째.

### 저장 후
병합/신규 저장된 노트에 대해 **13장 인덱싱 파이프라인** 실행.

## 13. 인덱싱 파이프라인 (저장·distill 후)

노트가 생기거나 바뀌면 **그 노트만 부분 재인덱싱**:
```
본문 변경
 → 청크 분할        (문단/헤딩 경계로 자르고, 너무 길면 ~수백 토큰 단위로 추가 분할)
 → 그 노트의 기존 chunks 행 삭제 (embedding·bigm/hnsw 인덱스는 행과 함께 자동 정리)
 → 새 청크 INSERT
 → 각 청크 임베딩: e5 prefix "passage: " 붙여서 → 같은 행 embedding 컬럼에 UPDATE/INSERT
 → 본문 [[링크]] 파싱 → 그 노트의 edges 재생성 (dst_note 해석 포함)
 → (신규 노트면) 같은 제목을 dst_title로 가진 미해결 edges의 dst_note 채움
```

**필수 디테일** (기존 코드 안 보고 개발하므로 명시)
- **e5 prefix**: 청크 임베딩엔 `passage: `, 검색 쿼리엔 `query: ` 를 **반드시** 붙인다. (multilingual-e5-large 규약 — 안 붙이면 검색 품질이 통째로 무너짐)
- **배치 임베딩**: 청크들을 한 번에 묶어 임베딩. 특히 distill은 노트 여러 개 → 청크 수십 개 → 1개씩 호출하면 느림.
- **부분 갱신**: 전체 재색인 아님. 바뀐 노트만.
- **동기 vs 비동기**:
  - 노트 1개 수동 저장 → **동기**(방금 저장한 게 바로 검색되는 게 자연스러움).
  - distill 다건 저장 → **FastAPI BackgroundTasks 비동기**(저장 직후 검색할 일 없음, 응답 멈춤 방지). 완료 전엔 그 노트만 검색에서 잠시 빠짐.
- **글자검색 인덱스**: `pg_bigm` GIN 인덱스(`idx_chunks_bigm`)는 행 INSERT/DELETE 시 **자동 유지** → 별도 트리거 불필요(sqlite-vec/FTS5 대비 단순화된 부분). 검색은 `text LIKE likequery(:q)`(GIN 가속) + `bigm_similarity(text, :q)` 랭킹.
- **트랜잭션 경계**: 한 노트 재인덱싱(청크 삭제→INSERT→임베딩 UPDATE→edges 재생성)은 **단일 트랜잭션**으로 묶어 부분 갱신 중 검색이 깨진 상태를 보지 않게 한다.
- **출처 안정성**: 재인덱싱 때 `chunk_id`가 바뀌므로 `message_citations`는 **note_id 기준으로 저장**(chunk_id 기준이면 재인덱싱마다 과거 출처가 끊김). 정밀 청크 인용은 향후.
- **이름변경/삭제 시 edges**: 링크가 제목 기준이라 노트 rename 시 다른 노트의 `[[옛 제목]]`이 깨짐. 정책: **깨진 채 미해결(dst_note=NULL)로 두기**. 노트 삭제 시 그 노트를 가리키던 edges는 dst_note만 NULL로(미해결).

### 14.3 FE가 호출하지 않는 것 (참고)
아래는 서버 내부 처리거나 인프라용 — **FE 코드에서 fetch 하지 않는다.**
- **내부 처리(엔드포인트 아님)**: 다시쓰기 · 노트 검색 · 대화 요약/압축 · distill 병합대상 매칭(청크-청크) · 인덱싱(청크·임베딩·edges) · 임베딩 모델 · openclaw 호출 → 모두 위 FE 엔드포인트(`/api/chat`·`/api/distill`·`/api/ingest`·`restore`) **내부에서 서버가 수행**.
- **인프라**: `GET /health` (모니터링/헬스체크).
- **브라우저 리다이렉트(fetch 아님)**: `/auth/callback/kakao`, `/auth/callback/github` — OAuth 공급자가 호출, FE는 부르지 않음.
- **중복**: `GET /api/backlinks/{title}` 는 backlinks가 `/api/page` 응답에 포함되므로 FE엔 불필요(외부 직접링크용으로만 선택).
